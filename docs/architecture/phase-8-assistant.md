# Phase 8 architecture: optional read-only incident assistant

## Scope and authority

The assistant answers questions only from evidence supplied in a request. It does not query
PostgreSQL or Redis, receive agent credentials, call deployment APIs, use model tools, or expose
an execution endpoint. It is absent from the default Compose startup and binds to host loopback
only when the `ai` profile is selected.

```text
operator -> assistant API -> redact -> provider (offline by default)
                              |                  |
                              +<- strict JSON ---+
                                      |
                         validate every citation
                                      |
                         cited analysis + proposals
                                      |
                         explicit human review receipt
                                      |
                              no execution path
```

The provider interface contains one operation: analyze redacted evidence. The optional OpenAI
adapter calls the Responses API with `store=false`, an empty tools list, a timeout, bounded retries
with exponential backoff and jitter, and a strict JSON schema. Provider output is treated as
untrusted and redacted again before validation.

## Safety controls

1. Evidence is bounded, assigned immutable request-local IDs (`E1`, `E2`, ...), and redacted before
   any provider receives it.
2. Claims and proposal rationales must cite supplied evidence IDs. Unknown citations cause the
   entire response to fail closed into abstention.
3. Missing evidence, malformed provider output, provider failures, and unsupported claims produce
   an explicit abstention with no proposal.
4. Every remediation is labeled `requires_human_approval`. A review receipt may approve or reject
   it, but the receipt states `execution_available=false` and triggers no callback.
5. The in-memory review registry is bounded. It is demo audit state, not durable incident state;
   a future durable implementation must remain isolated from operational write credentials.

## Phase acceptance criteria

- Default FleetPulse Compose startup has no assistant container and no AI dependency.
- Offline assistant and golden-set evaluation work without a network or API key.
- Common bearer tokens, assignments, database URL passwords, and private-key blocks are removed
  before provider input and API output.
- Every non-abstaining claim and proposal resolves to supplied evidence.
- Invalid citations and unavailable/invalid providers force abstention.
- Proposals cannot be reviewed unless generated by the current service process; duplicate reviews
  are rejected.
- The HTTP surface contains no execution or remediation endpoint.
- Optional container binds only to `127.0.0.1` and has no data-network access or operational secret.
- Evaluation output is preserved and labeled as deterministic accuracy/safety evidence, not a
  production-scale or general model-accuracy claim.

## Provider API basis

The adapter follows the official
[OpenAI Responses API](https://developers.openai.com/api/reference/resources/responses/methods/create)
and [Structured Outputs](https://developers.openai.com/api/docs/guides/structured-outputs)
contract checked on 2026-07-17. The model is environment-configurable because model availability
changes over time. No live-provider result is required for Phase 8 acceptance.
